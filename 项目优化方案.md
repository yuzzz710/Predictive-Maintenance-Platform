## 二、P0-1：运筹优化升级（回应评委核心建议）

### 2.1 现状诊断

当前维护计划的决策链路是**规则驱动**的，存在可优化空间：

| 模块 | 当前方法 | 不足 |
|------|---------|------|
| 停机优化器 (downtime_optimizer.py) | 6规则决策树 (if-else) | 硬编码阈值，无全局最优 |
| 工单排序 (strategy_selector.py) | 风险评分降序 + 截断 (max_orders) | 未考虑资源约束下的组合优化 |
| 备件规划 (spare_parts_planner.py) | 故障类型→备件映射 | 无库存优化模型 |
| 技师分配 (technician_assigner.py) | 12条规则匹配 | 未做负载均衡优化 |
| 维护日历 (成果文档 §6.3.5) | 贪心+回溯排程 | 未用数学规划模型 |

**问题本质**：当前系统回答了"哪些设备需要修"，但没有在数学上严格回答"在有限资源下，修哪些、什么时候修、谁去修能使总成本最小/总产出最大"。

### 2.2 优化方案：三层次运筹模型

#### 2.2.1 第一层：工单选择 — 风险削减背包问题

**问题定义**：
- 设N台设备进入预警状态，每台设备i有一个维护工单
- 工单i的参数：执行成本c_i（技师工时+备件+停机损失），风险削减量r_i（维护前综合风险−维护后残余风险）
- 工厂当周维护预算上限B，最大可用工时H
- **目标**：在B和H约束下，选择工单子集S，最大化总风险削减Σr_i

**数学模型**（0-1背包变体）：

```
max  Σ r_i · x_i           (最大化风险削减)
s.t. Σ c_i · x_i ≤ B      (预算约束)
     Σ t_i · x_i ≤ H      (工时约束)
     x_i ∈ {0, 1}         (选/不选)
```

**在现有代码中的落地**：在 `strategy_selector.py` 的 `apply_strategy_thresholds()` 方法之后，增加 `optimize_work_order_selection()` 方法，使用 `scipy.optimize` 或手写DP求解。当前 `max_orders` 和 `max_budget` 字段已存在于 `StrategyConfig` 中但 `max_budget=0` 表示未启用——补全即可激活。

**演示价值**：在仪表盘sec6展示"选中的工单 vs 因预算/工时约束延期的工单"，可视化约束如何影响决策。

#### 2.2.2 第二层：维护排程 — 带时间窗的多机调度

**问题定义**：
- 已选中K个工单需要安排到未来T天（建议T=14天滚动窗口）
- 每天有可用工时H_d（工作日=64h，周末=40h）
- 工单i的关键属性：所需工时t_i、时间窗[earliest_i, deadline_i]（基于健康分退化投影）、技师类型要求type_i
- 部分工单可合并（同技师类型+同设备区域+时间窗重叠）
- **目标**：最小化总拖期惩罚（Σ weight_i × tardiness_i），其中weight_i ∝ 设备成本风险

**数学模型**（带时间窗的单机批调度，Single Machine Batching with Time Windows）：

```
min  Σ w_i · T_i                                    (最小化加权拖期)
s.t. Σ t_j · y_{jd} ≤ H_d     ∀d               (日工时约束)
     Σ x_{id} = 1              ∀i               (每工单安排一次)
     x_{id} ≤ M · y_{id}       ∀i,d             (逻辑约束)
     T_i ≥ s_i + t_i - deadline_i                 (拖期计算)
     T_i ≥ 0                                       (非负拖期)
```

**在现有代码中的落地**：新建 `skills/predictive-maintenance-decision/scripts/maintenance_scheduler.py`，可用 `pulp` 或 `ortools` 求解（均为pip安装的轻量库，<5MB）。替换当前 `downtime_optimizer.py` 中基于6条if-else规则的调度逻辑。

**演示价值**：在仪表盘sec6新增"30天维护日历"的甘特图视图（利用ECharts甘特图），展示优化后的排程结果，对比规则排程vs优化排程的总成本差异。

#### 2.2.3 第三层：备件库存 — (s, S) 随机库存策略

**问题定义**：
- 每种备件j的需求D_j服从以故障频率为均值的泊松分布
- 单位持有成本h_j（资金占用+仓储），单位缺货成本p_j（紧急采购溢价+停机等待损失）
- 采购提前期L_j（天），订货成本K_j（固定成本/次）
- **目标**：确定再订货点s_j和订货量S_j，使期望总成本（持有+缺货+订货）最小

**(s, S)策略公式**（经典库存论）：

```
s_j = μ_j · L_j + z_α · σ_j · √L_j     (再订货点 = 提前期需求均值 + 安全库存)
S_j = s_j + √(2K_j·μ_j/h_j)             (目标库存 = s + EOQ)
```

其中z_α为服务水平参数（策略联动：成本效率α=0.85→z=1.04，生产效率α=0.92→z=1.41，质量优先α=0.98→z=2.05）。

**在现有代码中的落地**：新建 `skills/predictive-maintenance-decision/scripts/inventory_optimizer.py`，替换当前简单的故障类型→备件映射逻辑。输出库存策略表（每备件的s/S值+安全库存+ROP），同步至 `spare_parts_plan.csv`。

**演示价值**：在库存管理页面（`/inventory`）新增"库存优化仪表盘"：当前库存vs优化目标库存的对比柱状图、缺货风险Top 5、采购建议列表。

#### 2.2.4 三策略的多目标帕累托前沿

**问题定义**：
- 三种维护策略（成本效率/生产效率/质量优先）本质上是在三个目标函数间选择不同权重：
  - f₁：总维护成本（希望最小）
  - f₂：总停机时间（希望最小）
  - f₃：质量缺陷风险（希望最小）
- 当前三种策略是预设的固定参数（阈值/权重/SLA），缺少对"为什么是这些参数"的数学解释

**优化方案**：对每个策略生成帕累托前沿
- 用ε-约束法（ε-constraint method）求解：固定一个目标为约束，优化另一个目标
- 在前端展示三个策略在帕累托前沿上的位置，使用ECharts三维散点图（x=成本, y=停机, z=质量风险）
- 支持策略参数滑块：用户拖动权重滑块，实时看到工单选择和解的变化

**在现有代码中的落地**：新建 `skills/predictive-maintenance-decision/scripts/pareto_optimizer.py`，在 `strategy_selector.py` 中增加 `generate_pareto_frontier()` 方法。前端在仪表盘sec6的策略选择器旁边增加"策略分析"按钮，点击弹出模態窗口展示帕累托图。

**演示价值**：这是路演中最具视觉冲击力的运筹优化展示——三维散点图+实时滑块+三个策略的高亮定位，直观传达"系统在做数学优化"的信息。

## 三、P0-2：路演演示效果增强

### 3.1 现状诊断

当前演示能力：
- 故障注入6步演示（~60秒闭环）— **核心亮点，保持**
- 仪表盘8Tab — **内容多但导航分散**
- 10×10设备健康网格 — **视觉好但静态**
- 交互式KDE数据天花板 — **技术深度好但节奏慢**

路演问题：
1. 缺少一条"黄金演示路径"串联所有亮点
2. 6步故障注入是唯一有节奏控制的演示，其他页面纯浏览
3. 首页信息密度过高（10×10网格+2×5卡片+工单区+备件表），评委视角难聚焦
4. 角色切换需要reload，打断演示连贯性

### 3.2 优化方案

#### 3.2.1 "黄金演示路径"设计

设计一条7分钟的完整演示路线，评委无需操作即可看到全部核心能力：

```
[30s] 角色门(role-gate) → 选运维工程师 → 首页10×10网格全览
  ↓
[60s] 首页 → 点击红色高危设备CNC_042 → 右侧SHAP归因面板滑出
  ↓
[90s] 首页 → 点击"⚡故障注入演示" → 选CNC_085+Type4+中度 → 6步自动演示（每步调为1.0s加速）
  ↓
[60s] 仪表盘sec4 → 算法对比实验墙 → 7种算法AUC柱状图+随机线0.5+可用线0.70
  ↓
[60s] 仪表盘sec2 → KDE数据天花板交互演示 → 拖动滑块从Phase0→3，Youden's J 0.075→0.90
  ↓
[60s] 仪表盘sec6 → 维护策略选择器（三策略对比）+ 帕累托前沿图（优化后新增）
  ↓
[60s] 仪表盘sec7 → 传感器升级ROI柱状图+三阶段投资卡片
  ↓
[30s] Chat → AI Copilot快速问答："当前哪些设备风险最高？"
```

**实现方式**：新增"演示模式"按钮（首页Header，路演专用），点击后自动按时间轴推进，每步高亮目标区域+暗化其他区域（聚光灯效果），评委仍可打断进行自由互动。

#### 3.2.2 首页视觉聚焦优化

- **减少一屏信息量**：10×10网格从Keycap 3D风格改为扁平彩色矩阵（更快渲染、更清晰），2×5紧急设备网格改为横向滚动条带
- **高亮优先级设备**：首页自动将Top 3风险设备（按当前策略）用脉冲动画+聚光边框突显
- **KPI卡片动画化**：健康/退化/高危设备数从静态数字改为入场计数动画（数字从0滚动到目标值,~1.5s），增加动态感

#### 3.2.3 故障注入演示加速优化

- 6步间隔从1.5s缩短为1.0s（总时长60s→45s）
- 每步增加"数据变化动画"：Z-Score数值跳动、健康分进度条缩短、工单卡片从右侧飞入
- 在Step 5（技师分配）中增加邮件通知的模拟Toast弹出

#### 3.2.4 角色切换零刷新

当前角色切换调用 `window.location.reload()`，演示时造成2-3秒白屏。优化方案：
- `RoleSwitcher.set(role)` 改为动态切换 `data-role` 属性（不reload）
- CSS过滤自动生效（`:not()`伪类即时响应属性变化）
- 需要重新加载的数据（如角色特定的工单图表），通过API异步请求更新而非全页刷新
- 角色切换动画：当前页面淡出→淡入新视图（~300ms过渡）

#### 3.2.5 全局状态栏（路演HUD）

在首页底部增加一条半透明状态栏（仅在检测到`?demo=true` URL参数时显示）：
```
◆ 系统状态: FULL | 监控设备: 100台 | 告警精确率: 83.9% | 当前策略: 生产效率 | 活跃工单: 20 | 月度节省: $127,500
```
HUD数据每秒从 `degradation_status.json` + `strategy_comparison.csv` + `industrial_maintenance_plan.csv` 聚合刷新，给评委持续传递"这是一个实时运行的系统"的印象。

## 四、P2：AI应用能力微调

### 4.1 MCP+Agent Skills 叙事强化

当前25个Gateway Tools已实现MCP协议标准化，但评委在文档和演示中可能难以感知"为什么MCP是创新点"。

**优化方案**：
- 在 `technical-overview.html` 板块05增加"MCP协议 vs 传统API调用"对比动画：传统方式（每个工具单独写HTTP端点→LLM通过prompt描述理解工具）vs MCP方式（标准化工具描述→服务端自动发现→LLM直接理解JSON Schema）
- 在Agent Orchestrator中增加"Agent自主决策日志"输出：记录每次对话中LLM自主决定调用了哪些工具、调用顺序、是否进行多轮迭代，在Chat界面底部以可折叠的"决策日志"展示（`/chat`页面右下方小按钮）

### 4.2 RAG降级演示

RAG的三级嵌入降级（BGE→DeepSeek API→TF-IDF）是一个强有力的工程可靠性亮点，但当前只在后台自动执行，演示时不可见。

**优化方案**：
- 在Chat界面增加"RAG状态指示器"（小图标）：当前使用的嵌入层级+响应延迟
- 在 `knowledge-base.html` 增加"降级模拟"按钮：手动触发降级，展示从BGE（本地15ms）→DeepSeek API（网络300ms）→TF-IDF（本地5ms）的切换过程

## 七、风险提示

1. **运筹优化求解器依赖**：`pulp` 和 `ortools` 均为纯Python包，pip安装<30秒。但若决赛环境不允许联网安装，需提前在requirements.txt中添加并预装。
2. **演示模式复杂度**：聚光灯效果和自动推进涉及较复杂的DOM操作和时序控制，需充分测试在评委可能打断/跳过的场景下的状态恢复。
3. **优化模型数据依赖**：运筹优化需要准确的成本参数（工时、备件价格、停机损失），当前数据中的成本参数来自CSV，需确保这些参数在演示数据中保持一致性和合理性。
4. **不要过度优化**：评委看到的是一次性演示，不是生产系统。优化模型的计算结果应预计算并缓存到CSV/JSON——演示时直接加载结果，而不是实时求解优化问题（避免求解器卡顿影响演示节奏）。
