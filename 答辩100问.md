## 一、AI应用能力 — 模型环境（10%）~18问

### Q1：你们用的是什么大模型？为什么选它？

我们使用的是 DeepSeek 大模型，通过 API 接入。选择 DeepSeek 的原因：一是中文理解能力强，工业运维场景涉及大量中文术语和文档，DeepSeek 的中文能力有优势；二是性价比高，适合工业场景的持续调用；三是支持 Function Calling（函数调用），我们的 AI Copilot 需要调用 25 个后端工具来查询设备状态、生成工单等，DeepSeek 原生支持这个能力。

### Q2：大模型在你们系统里具体干了什么事？

大模型承担三个角色：第一，**AI 运维助手** — 用户用自然语言问"CNC_023 最近怎么样？"大模型理解意图后调用工具查询数据，再用通俗语言回答；第二，**根因分析** — SHAP 算法找出故障贡献最大的参数后，大模型将其翻译成工业可读的诊断结论；第三，**报告生成** — 根据运维数据自动撰写周报、月报，包含数据摘要和行动建议。

### Q3：你们的开发环境是怎么搭建的？

我们的环境完全自主搭建：Python 3.10+ 作为主语言，FastAPI 做 Web 后端，DeepSeek API 做大模型接入，Chroma 向量数据库做知识检索，SQLite 做轻量级状态存储。所有依赖都写在 requirements.txt 里，一行 `pip install -r requirements.txt` 即可部署。前端是纯 HTML/CSS/JS，不依赖 Node.js 构建工具，打开即用。

### Q4：大模型如果没有 API 网络怎么办？有没有本地备选方案？

我们有四级降级机制。大模型不可用时，系统自动降级到规则引擎：AI 助手回退到话术库匹配（40+ 板块预生成话术），报告生成用模板填充，诊断结论直接输出 SHAP 数值+规则标签。此外，RAG 检索的嵌入模型 BGE-small 是本地部署的（约 100MB），完全不依赖网络。

### Q5：大模型调用有延迟吗？用户等待体验怎么样？

Copilot 对话采用 SSE（Server-Sent Events）流式传输，大模型一边生成一边推送给前端，用户看到文字逐字出现，类似 ChatGPT 的打字效果，感知延迟很低。工具调用（如查询设备状态）通常在 2-3 秒内完成。复杂任务（如跑完整流水线）会显示进度状态，用户不会对着空白页面等。

### Q6：你们怎么保证大模型不乱说（幻觉问题）？

三层约束：第一，**工具强制** — 查询类问题必须调用工具获取真实数据，禁止凭空编造数字；第二，**知识库 RAG** — 专业问题先检索知识库，把检索到的文档作为上下文喂给大模型，限定它基于文档回答；第三，**Prompt 约束** — 系统提示词明确要求"不知道就说不知道，不要编造"，并给出了具体的回答格式模板。

### Q7：模型环境这块，你们遇到过什么坑？怎么解决的？

最大的坑是 DeepSeek API 的 function calling 格式兼容性。OpenAI 的工具调用返回的是 `tool_calls` 数组，DeepSeek 早期版本返回格式略有不同。我们写了一个适配层 `deepseek_client.py`，统一处理两种格式，这样上层业务代码不用关心用的是哪家的模型。另外，Windows 环境下 UTF-8 编码是个持续的坑，所有文件读写我们都显式指定 `encoding='utf-8'`。

### Q8：你们系统支持换其他大模型吗？比如换成通义千问或者文心一言？

支持。我们在 `gateway/deepseek_client.py` 里做了统一的 LLM 客户端封装，换模型只需要改 `.env` 文件里的三个环境变量：`DEEPSEEK_API_KEY`、`DEEPSEEK_BASE_URL`、`DEEPSEEK_MODEL`。只要新模型兼容 OpenAI 的 API 格式和 function calling 协议，一行代码都不用改。

### Q9：AI 助手能处理多轮对话吗？还是只能一问一答？

支持多轮对话。前端维护完整的对话历史，每次请求把历史消息一起发给大模型。大模型可以记住之前提到的设备编号、故障类型，进行上下文连贯的追问。比如先问"CNC_023 最近有问题吗？"，接着问"那给它生成一个维修工单"，助手知道"它"指的是 CNC_023。

### Q10：你们的大模型调用有成本控制吗？

有。一是 Prompt 设计精炼，减少不必要的 token 消耗；二是工具调用设置了最大 5 轮循环，防止死循环；三是 RAG 检索先过滤再喂给模型，而不是把整个知识库丢进去；四是话术库优先命中（预生成内容零成本），只有命中不了才调用大模型。

### Q11：如果同时有 100 个人用 AI 助手，系统撑得住吗？

当前是单机部署，并发能力有限。但我们做了几个设计：轻量级 FastAPI 异步框架天然支持高并发；SQLite 读操作不加锁；大模型 API 调用是异步非阻塞的。如果要支撑更大规模，可以加 API 网关做限流、Redis 做会话缓存、数据库升级到 PostgreSQL，架构上已有预留。

### Q12：你们的模型接入方式是什么？直接调 API 还是本地部署？

采用云端 API 调用的方式。原因是我们的应用场景是实时交互的运维助手，需要低延迟响应，而本地部署大模型对 GPU 资源要求很高。通过 API 调用，我们只需要一个轻量级的 Python 后端即可，部署维护成本极低。

### Q13：除了 DeepSeek，你们的技术栈里还用了哪些"AI"技术？

除了大模型，我们还用了：**XGBoost 和 Multi-Task Neural Network** 做故障预测；**BGE-small-zh-v1.5 嵌入模型**做知识库向量检索；**SHAP（Shapley Additive Explanations）**做模型可解释性分析；**Z-Score 统计方法**做异常检测。是一个"统计+机器学习+深度学习+大模型"的多层次 AI 体系。

### Q14：大模型能直接做故障预测吗？还是只能当助手？

在我们的系统里，大模型不做预测，只做交互和解释。故障预测由专门的统计模型（Z-Score、T²）和机器学习模型（XGBoost、MTNN）完成。原因是：大模型是通用语言模型，没有在传感器时序数据上训练过，直接让它做数值预测准确率不可控。我们的设计原则是"让专业的模型做专业的事，大模型负责把它们的结果翻译给人看"。

### Q15：你怎么评估大模型回答的质量？

三个维度：一是**准确性** — 是否基于真实数据而非编造，可以对照后台数据验证；二是**完整性** — 是否覆盖了用户问题的所有要点；三是**可操作性** — 是否给出了明确的下一步建议而非泛泛而谈。在话术库中我们预设了 40+ 板块的标准回答，有明确的参考基准。

### Q16：大模型的 system prompt 是怎么设计的？

核心结构分四部分：角色定义（"你是工业运维专家，服务于 100 台 CNC 机床"）、能力边界（"你可以查询设备数据、生成工单、解释故障，但不能修改系统配置"）、工具使用规则（"查询数据必须调用工具，不能编造"）、回答格式要求（"使用中文，简洁专业，给出可执行的建议"）。

### Q17：未来有没有考虑做模型的微调（Fine-tuning）？

技术上是可行的。如果我们积累了足够多的真实运维对话数据（比如几千条高质量的问询-回答对），可以用 LoRA 等轻量级微调方法在 DeepSeek 上做领域适配，让模型更懂工业术语和运维流程。但目前阶段，Prompt Engineering + RAG + Tool Use 的组合已经能覆盖 90% 以上的需求。

### Q18：安全方面，怎么防止用户通过 AI 助手做恶意操作？

两层防护：第一，**工具白名单** — AI 助手可调用的工具都是只读查询或受控写操作（如生成工单），不能执行系统命令、修改代码或删除数据；第二，**角色权限** — 不同角色可调用的工具不同，操作员不能执行管理者专属的"修改维护策略"操作。系统在工具执行前会验证调用者的角色权限。

## 二、AI应用能力 — 模型应用（10%）~18问

### Q19：你们用软件工程方法解决了什么场景问题？

我们解决的场景是：**工厂里有 100 台 CNC 机床，运维团队每天要盯着 400 个传感器参数，判断哪些设备需要维修、什么时候修、怎么修。** 靠人眼根本看不过来，传统的定期维修又浪费钱。我们的系统把"数据采集 → 异常检测 → 根因诊断 → 维修决策 → 工单执行 → 修复验证"全流程自动化了。

### Q20：系统最终产出的可执行方案是什么样子的？

产出物是**一份包含优先级排序的维修工单**。每张工单包含：设备编号、故障类型和严重程度、SHAP 根因分析结论（哪个参数异常、对应什么工业问题）、建议维修动作、需要的备件清单、预估停机时间、推荐维修时段、指派的技术人员。工单会通过邮件自动通知到人，并有 15 分钟超时升级机制。

### Q21：你们的维护策略有三种，切换后具体会发生什么？

切换维护策略后，后端会**重新计算所有工单**。三种策略的区别在于决策公式里的权重不同：
- **成本效率优先**：优先修维修成本低、故障影响小的设备，目标是省维修费；
- **生产效率优先**：优先修对生产影响大的设备，目标是保产能；
- **质量优先**：优先修影响加工精度的设备，目标是保产品良率。

切换策略不是前端换个颜色，是后端用不同权重重新跑整个决策引擎。

### Q22：你们怎么证明系统真的有用？有没有量化指标？

有。我们跑了完整的回测验证：
- **告警精度**：83.9%（Z>2.0 时）
- **健康评分分布**：100 台设备中 10% 危急、88% 亚健康、2% 健康，符合真实工厂的"大部分设备带病运行"常态
- **七个算法对比**：在同样的 2999 行真实数据上训练和测试，公开了 AUC 从 0.517 到 0.589 的全部结果
- **ROI 预测**：传感器升级方案的投资回收期 5-8 个月，5 年 ROI 723%-1136%

### Q23：AI 预测到底准不准？你们诚实地说。

**在现有四个传感器参数下，纯 AI 预测不可靠。** 这是我们的核心发现。四个参数的 Youden's J 指数最高只有 0.075（电压），AUC 上限约 0.537，也就比抛硬币（0.5）强一点点。我们跑遍了 7 种主流算法（逻辑回归、SVM、随机森林、XGBoost、LightGBM、CNN、多任务神经网络），没有一个突破 0.59。所以我们诚实地说：**以现有传感器，无法实现可靠预测，需要升级传感器。**

### Q24：既然 AI 预测不准，那你们的系统还有什么用？

这正是我们的价值所在。第一，我们**用统计方法做异常检测**，Z-Score 的精度有 83.9%，这比 AI 预测可靠得多；第二，我们**量化了数据天花板**，告诉工厂"现有数据最多能做到什么程度，要更进一步需要投入什么"；第三，我们**把诊断和决策自动化了** — 即使预测不准，从告警到工单的全流程自动化仍然节省了 70% 以上的人工处理时间。

### Q25：健康评分是怎么算出来的？为什么选 8 个维度？

健康评分综合了 8 个维度：Z-Score 异常度、T² 统计量、成本风险等级、趋势斜率、历史故障频率、最近故障距今天数、SHAP 重要性加权、传感器漂移程度。之所以选 8 个维度而不是简单的阈值判断，是因为单一参数异常不一定是故障（可能是瞬态干扰），多维交叉验证才能提高置信度。这 8 个维度的选择参照了 ISO 13374 设备状态监测国际标准。

### Q26：SHAP 可解释性是什么？为什么重要？

SHAP 是一种"打开 AI 黑盒"的技术。普通 AI 模型告诉你"这台设备 80% 概率会坏"，但不告诉你为什么。SHAP 能分解出每个传感器参数对预测结果的贡献值，比如"电压贡献了 62%、温度贡献了 28%、转速贡献了 10%"。在工业场景，**"为什么"比"是什么"更重要** — 维修工程师需要知道是电源模块还是散热系统出问题，才能采取正确的维修动作。

### Q27：AI Copilot 和 ChatGPT 有什么区别？

ChatGPT 是通用对话 AI，不了解你的工厂、不知道你的设备状态、也不会帮你生成工单。我们的 AI Copilot 是**深度嵌入业务系统**的：它能实时查询 CNC_023 的真实传感器数据，能检索设备维修手册，能帮你一键生成维修工单并邮件通知工程师。它不是聊天机器人，是**有执行能力的运维助手**。

### Q28：系统能自动发现新型故障吗？

当前系统的故障检测基于已知的 9 种故障类型的特征模板，新出现的故障模式如果不在模板库中，系统会标记为"未知异常"并告警，但无法给出具体的故障类型诊断。这是一个合理的工程权衡 — 工业场景中 95% 的故障是已知类型，优先把已知问题解决好。未来可以引入聚类分析对"未知异常"进行模式发现。

### Q29：系统上线后，怎么持续优化模型？

我们设计了反馈闭环：维修完成后，工程师需要确认"故障类型判断是否正确、维修方案是否有效"，这些确认数据回流到故障案例库。随着案例积累，模板库自动丰富，SHAP 特征权重持续更新，决策引擎的推荐策略也会基于真实维修效果做调整。

### Q30：你们提到了四种降级模式，这是为了应对什么情况？

工业级系统不能因为某个模块挂了就整体瘫痪。我们的四级降级设计保证**任何条件下系统都能产出可执行的维修建议**：
- **FULL**：完整功能，统计+ML+SHAP 全开
- **STAT_ONLY**：ML 服务挂了，用纯统计方法，~9 秒可跑完
- **RULE_ONLY**：统计也挂了，用预设规则引擎，~3 秒
- **EMERGENCY**：数据都读不到了，基于传感器实时波动值发出告警，<1 秒

导航栏上的圆点指示灯让运维团队一眼就能看到当前系统运行在哪个级别。

### Q31：故障注入演示是怎么做的？改了真实数据吗？

不改。故障注入是**纯内存计算**：前端选择设备和故障类型后，后端在内存中按故障特征模板修改传感器数值，跑一遍完整的"检测→诊断→决策→工单"流程，把过程展示给观众看。演示结束后数据自动丢弃，数据文件毫发无损。这是专门为路演和评委演示设计的功能。

### Q32：你们的三种角色是怎么分的？需要登录吗？

不需要登录系统。我们采用了**基于角色的页面过滤机制**：在入口页面选择角色（操作员/管理者/开发者）后，同一套 HTML 页面通过 CSS 自动显示或隐藏不同模块。操作员看到的是设备监控和工单执行，管理者看到的是 KPI 看板和策略选择，开发者看到的是全部数据和技术细节。这种方式部署简单，适合演示环境。

### Q33：开发者和普通用户看的有什么不同？

开发者多了四个视图：**数据探索面板**（原始传感器数据可视化）、**算法对比结果**（7 种算法的 AUC 柱状图）、**模型评估详情**（训练集/测试集的混淆矩阵和 ROC 曲线）、**系统架构总览**（DAG 流水线状态和组件健康状况）。这些是运维人员不需要关心的底层细节。

### Q34：设备网格（Device Grid）的颜色代表什么？

10×10 网格中每个方格代表一台 CNC 设备，颜色从绿色（健康）→ 黄色（亚健康）→ 橙色（警告）→ 红色（危急），根据综合健康评分映射。点击任意方格可以钻取到该设备的详细传感器趋势、SHAP 诊断报告和最近的工单历史。网格设计让 100 台设备的状态一目了然，类似服务器机房的监控大屏。

### Q35：3D 球体数字孪生是干什么用的？

这是展厅演示功能。100 台设备以球面分布的方式在三维空间中展示，设备健康评分映射为球体的颜色和大小。支持手势识别（握拳旋转、张开缩放、双指平移），也可以用鼠标操作。目的是给评委和参观者一个**直观、震撼的视觉效果**，让他们快速理解"我们在监控 100 台设备"这件事。

### Q36：系统每天自动做什么事情？

每天早上 6:00：自动跑健康检查流水线，对 100 台设备逐一评估，健康评分低于 40 分的自动生成告警工单并邮件通知；每周一早 7:00：自动生成运营周报（包括设备健康趋势、维修工单统计、备件消耗、成本分析）并邮件发送；每 15 分钟：检查是否有超时未处理的工单，自动升级通知。

## 三、数据处理能力 — 数据获取（10%）~18问

### Q37：你们的数据从哪里来的？

数据来自一个真实的工业数据集，包含 4 个脱敏文件：**LOG**（设备运行日志，时序传感器数据）、**SUMMARY**（设备汇总统计）、**ASSEMBLY**（装配关系数据）、**TESTS**（测试记录）。数据覆盖 100 台 CNC 机床，4 个传感器参数，9 种故障类型。虽然数据是脱敏的，但数据结构和分布特征完全模拟真实工厂环境。

### Q38：数据量有多大？

核心的 LOG 文件约 2999 行有效数据（经过清洗后），每行包含设备 ID、时间戳、4 个传感器读数和故障标签。对于工业场景来说不算大，但这恰恰反映了真实问题 — 很多工厂就是数据不足，这也是我们强调"数据天花板"和"传感器升级"的现实依据。

### Q39：你们对数据做了哪些加工处理？

五步处理流程：**清洗**（去除异常时间戳和空值）→ **标注**（基于故障日志为每条记录打故障标签）→ **基线构建**（为每台设备建立独立的参数基线，≥3 样本才建基线）→ **特征工程**（提取 Z-Score、T²、趋势斜率等衍生特征）→ **数据集划分**（按设备分组避免数据泄露，确保训练集和测试集的设备不重叠）。

### Q40：为什么要给每台设备单独建基线？不能用全局阈值吗？

因为**设备之间的差异太大了**。我们的方差分解分析显示，61-73% 的数据波动来自设备间差异，而不是时间上的退化。这意味着 CNC_001 正常运行的电压范围可能和 CNC_050 完全不同。用全局阈值会导致两类问题：一部分设备疯狂误报，另一部分设备坏了都不告警。逐设备基线是强制性的工程要求。

### Q41：如果设备样本太少怎么办？

三层回退策略：
- **≥6 个样本**：直接使用该设备自身的历史数据建独立基线；
- **3-5 个样本**：用该设备数据与所属簇的聚类中心做加权混合（自身 60%+簇 40%）；
- **<3 个样本**：完全使用所属聚类的簇中心作为初始基线，随数据积累逐步过渡到独立基线。

### Q42：数据标注是怎么做的？人工标注还是自动标注？

半自动标注。原始数据中有故障日志文件（记录了每次故障的设备、时间、类型），我们写脚本自动将传感器数据与故障日志按时间窗口关联，打上故障类型标签。标注完成后人工抽样验证了标注准确性。故障案例库的 833 条记录也是自动从 LOG 数据中提取和组织的。

### Q43：训练集和测试集怎么分的？有没有数据泄露风险？

按设备分组划分，**不按行随机划分**。这是关键设计：同一台设备的所有数据要么全在训练集、要么全在测试集。如果按行随机分，CNC_023 的前 100 行在训练集、后 50 行在测试集，模型就能"作弊"学到这台设备的特性，AUC 会虚高。按设备分组后测出来的 AUC 才是真实泛化能力。

### Q44：数据存储用的是什么？为什么不用 MySQL 之类的数据库？

当前主要使用 CSV 和 JSON 文件存储，工单状态用 SQLite。选择的原因是：第一，数据集规模不大（千行级别），文件读写已足够快；第二，CSV/JSON 方便评委直接打开查看，透明可审计；第三，零配置部署，不需要安装数据库服务。SQLite 用于需要事务支持的状态机（工单流转），比文件锁可靠。未来规模扩大可平滑迁移到 PostgreSQL。

### Q45：前端仪表盘上的数据是实时的吗？

当前是"准实时"模式。流水线运行后将结果同步到 `web-dashboard/data/` 目录下的 CSV/JSON 文件，前端通过 API 读取。在演示环境中，我们通过定时任务（每 30 分钟跑一次流水线）和手动触发（Copilot 可以调用运行流水线的工具）来保持数据新鲜度。完整的实时数据流需要 MQTT/OPC-UA 等工业协议对接，是后续扩展方向。

### Q46：你们怎么处理数据质量问题？比如传感器读数突然跳变。

跳变检测已经内建在统计模块中：当单个数据点偏离基线超过 5 个标准差时，标记为"疑似传感器异常"而非设备故障。区分传感器故障和设备故障很重要 — 修传感器和修设备是完全不同的动作。如果连续 3 个点都异常，则升级为设备故障告警。

### Q47：数据是持续积累的吗？系统支持增量更新吗？

流水线支持全量和增量两种模式。首次运行做全量计算（从零开始建立所有基线），后续可以只对新数据做增量更新（追加 Z-Score 计算、更新健康评分）。DAG 调度器能识别哪些步骤需要重跑、哪些可以用缓存结果，避免重复计算。

### Q48：如果数据文件被误删了，系统还能运行吗？

能，降到 EMERGENCY 级别。此时系统不依赖任何历史数据，仅基于传感器实时波动发出告警。虽然诊断精度下降，但不会"瞎"了 — 基本的异常感知能力还在。这就是四级降级设计的实际价值。

### Q49：你们的数据处理流程能复用到其他类型的设备吗？比如注塑机或冲压机？

核心流程是通用的：数据清洗 → 基线构建 → 异常检测 → 故障诊断 → 决策推荐。如果换成其他设备类型，需要调整的主要是：传感器参数名、故障类型定义、SHAP 特征映射、维修策略模板。约 70% 的代码可以直接复用，30% 需要做领域适配。这正是模块化 Pipeline 设计的优势。

### Q50：你们有没有做过数据增强？工业数据通常很少。

做了。对于样本量不足的故障类型（如某些罕见故障只有几十条记录），我们使用了 SMOTE（合成少数类过采样）技术生成合成样本，平衡训练数据。但我们很克制 — 只在训练集中做增强，测试集保持原始分布，确保评估结果可信。

### Q51：数据的时效性怎么考虑？三个月前的数据和昨天的数据权重一样吗？

不一样。基线计算中，近期数据权重更高（指数衰减加权），今天的数据比三个月前的数据更能反映设备当前状态。趋势分析也区分短期趋势（最近 24 小时）和长期趋势（最近 30 天），不同时间尺度捕捉不同的退化模式。

### Q52：你们的数据集能公开吗？有没有隐私问题？

数据集已经是脱敏的 — 设备用 CNC_001~100 编号，没有工厂名称、地理位置等隐私信息，传感器数值也做了归一化处理。如果学校或比赛组织方需要验证，我们可以提供数据集和完整的处理脚本。

### Q53：你是怎么定义"正常"和"异常"的？

基于 3-sigma 统计原则：设备在正常运行期间的数据服从正态分布，我们取 ±3 个标准差作为正常范围（覆盖 99.7% 的正常波动），超出范围即为异常。但单点异常不等于故障，需要结合持续时间、多参数交叉验证、趋势方向综合判断。

### Q54：你们用什么工具做数据可视化？

前端全部使用 **ECharts**（Apache 开源项目），是业界最强的数据可视化库之一。包括：时序折线图（传感器趋势）、柱状图（算法对比）、热力图（设备健康矩阵）、仪表盘（健康评分）、散点图（KDE 分布）、瀑布图（SHAP 特征贡献）、旭日图（降级状态）、雷达图（多维健康评分）等 10+ 种图表类型。

## 四、数据处理能力 — 数据应用（10%）~10问

### Q55：你们说 AI 预测不准，具体怎么衡量的？

用六个标准指标在真实测试集上评估：**AUC**（区分故障和正常的能力）、**精确率**（告警中有多少是真的）、**召回率**（故障中有多少被发现了）、**F1 分数**（精确率和召回率的调和平均）、**Youden's J 指数**（综合灵敏度和特异度）、**混淆矩阵**（分类型统计预测对了多少、错了多少）。

### Q56：Youden's J 指数是什么意思？为什么一直在提它？

Youden's J 指数 = 灵敏度 + 特异度 - 1，取值范围 0 到 1。0 表示完全没有区分能力（跟瞎猜一样），1 表示完美区分。它是医学诊断中评估检测方法有效性的金标准，我们把它引入到工业故障检测。J<0.3 通常认为该方法缺乏实际应用价值，而我们四个参数的 J 值都在 0.075 以下，说明单靠现有传感器做预测确实不够。

### Q57：七个算法对比的结果说明了什么？

说明了两件事：第一，**深度神经网络（MTNN 0.589、CNN 0.547）确实比传统方法（逻辑回归 0.523）略好**，但提升幅度很小（仅 0.066），说明问题的瓶颈在数据质量而非模型复杂度；第二，**所有方法都在 0.6 以下**，都达不到可靠应用的标准。这证明我们需要多信号融合 + 传感器升级，而非"换个更强的模型"。

### Q58：83.9% 的告警精度是怎么算出来的？

在 Z>2.0 的阈值下：系统发出的所有告警中，有 83.9% 确实对应真实的设备故障，16.1% 是误报。这是用历史数据回溯验证的结果。我们选择这个阈值是在"漏报"（有故障没告警）和"误报"（没故障瞎告警）之间做的平衡 — 工业场景更怕漏报，所以适当容忍误报。

### Q59：你们有没有做 A/B 测试或者和人工对比？

在算法对比实验中，我们以故障日志中记录的真实维修时间作为 Ground Truth（基准真相），对比了 7 种算法的预测结果。另外，决策引擎生成的维修工单（优先级排序、备件推荐）与故障案例库中的实际维修方案进行了交叉比对，验证推荐的合理性。

### Q60：KDE 分布图在前端有什么用？

KDE（核密度估计）分布图展示了四个传感器参数在"正常"和"故障"两种状态下的概率分布。用户拖动滑块可以直观看到两个分布的重叠程度 —— 重叠越严重，说明这个参数越难区分正常和故障。这是一个**教学和说服工具**：让评委和管理者一目了然地理解为什么现有数据不够用。

### Q61：系统运行的结果你们有统计分析吗？

有。每次流水线运行都会产出统计摘要：告警数量/类型分布、健康评分分布直方图、Top 10 高危设备排名、故障类型频率、维修工单完成率、平均工单处理时长等。这些汇总数据驱动了管理看板上的 KPI 指标。

### Q62：你们怎么评估 SHAP 解释的准确性？

两层验证：第一，**工业一致性检查** — SHAP 给出的 Top 贡献参数是否与该故障类型的物理机理一致（比如轴承磨损应该对应振动信号异常，而不是温度）；第二，**后验证** — 维修完成后，由维修工程师反馈"根因分析是否正确"，反馈结果回流到模型。

### Q63：你们的数据分析结果有没有发现什么反直觉的结论？

有。最反直觉的是**温度参数在区分故障时的表现比预期差很多**（Youden's J 仅 0.067，排第三）。直觉上认为"设备出问题会发热"，但数据表明：CNC 机床的温控系统很成熟，故障初期的温度变化非常微弱，要等到故障严重到一定程度才明显升温，那时已经太晚了。这个发现支持了我们的传感器升级建议（优先加振动传感器而非温度传感器）。

### Q64：你们的"成本风险矩阵"是怎么算的？

综合考虑三个因素：**故障发生概率**（来自历史频率和当前健康评分）、**故障影响程度**（停机时长 × 设备产能权重）、**维修成本**（备件费用 + 人工费用 + 停机损失）。三者相乘得到风险值，按风险值排序确定维修优先级。这不是简单的"谁坏了先修谁"，而是量化的风险管理。

## 五、技术创新能力 — 工具创新（10%）~12问

### Q65：你们用到了哪些开源工具或创新技术？

核心技术栈全部基于开源：Python、FastAPI、XGBoost、PyTorch、SHAP、Chroma 向量数据库、ECharts、Three.js、BGE 嵌入模型、DeepSeek 大模型。创新技术方面，我们使用了 MCP（Model Context Protocol）协议来组织工具调用，设计了 Agent Skills 技能包体系来模块化管理流水线，还自研了 DAG 调度器做并行任务编排。

### Q66：MCP 协议是什么？你们怎么用的？

MCP（Model Context Protocol）是 Anthropic 推出的一个开放协议，用于标准化 AI 模型与外部工具的交互。我们的 25 个 Gateway 工具全部遵循 MCP 的 Tool 定义格式（name/description/parameters JSON Schema），这样任何支持 MCP 的 AI 客户端都能直接调用我们的工具。我们选择这个协议是为了**架构的前瞻性和互操作性**。

### Q67：Agent Skills 是什么？和普通函数有什么区别？

Skills 是我们设计的技能包体系，每个 Skill 是一个自包含的模块（有独立的 SKILL.md 文档、scripts/ 目录、输入输出契约），五个 Skills 串联成 DAG 流水线。和普通函数的区别在于：每个 Skill 有自己的**降级策略**（如果这个 Skill 失败了，下一个 Skill 应该怎么处理）、**缓存策略**（输入没变就复用上次结果）、**版本管理**（SKILL.md 记录了变更历史）。

### Q68：DAG 流水线是怎么设计的？

DAG（有向无环图）调度器定义了 5 个 Skill 的执行依赖：data_prep 必须最先执行；stat_inference 和 ml_inference 在 data_prep 之后可以并行执行（这两个互不依赖）；diagnosis 等 stat 和 ml 都完成后汇总执行；decision 等 diagnosis 完成后执行。调度器使用 Python 的 ThreadPoolExecutor 实现并行，最大化利用 CPU。

### Q69：为什么选择 DAG 而不是简单的顺序执行？

因为 stat_inference 和 ml_inference 是独立的，可以并行跑。如果顺序执行需要 18 秒，并行只需要 11 秒（以慢的那个为准）。虽然节省的绝对值不大，但这个架构设计展示了**工程思维**：当我们加更多并行 Skill（比如加一个振动分析 Skill）时，只要不依赖其他 Skill，就不会增加总时间。

### Q70：你们自己写了哪些代码，哪些是调库？

**完全自研的部分**：整个流水线调度器（DAG + 降级管理）、决策融合引擎（8 维健康评分公式、三种策略的加权逻辑）、SHAP 根因映射（SHAP 库给出特征重要性，但将特征值翻译为工业根因的逻辑是我们自己写的）、话术库系统、角色权限 CSS 过滤机制、故障注入引擎。

**基于开源库的部分**：XGBoost/MTNN 模型训练、SHAP 值计算、向量检索、大模型 API 调用、前端图表渲染。

**大致的自研比例**：后端业务逻辑约 70% 自研，前端展示约 50% 自研。

### Q71：你们觉得整个项目中最有技术创新的点是哪个？

**"数据天花板量化 + 传感器投资路线图"这个闭环。** 大多数 AI 项目会回避"模型不准"这个问题，或者用各种技巧强行提升数字。我们选择直面它：用 Youden's J 量化数据能力上限，用 KDE 可视化展示为什么不准，然后用三阶段传感器升级方案给出解决路径。这体现了**技术诚实和工程务实**的结合。

### Q72：向量数据库为什么选 Chroma 而不是其他的？

Chroma 有三个优势：第一，**嵌入式部署**，不需要单独的服务进程，一个 `pip install` 就搞定，和我们的零配置理念一致；第二，**Python 原生 API**，和我们技术栈无缝衔接；第三，**足够应对规模**，我们的知识库只有 1400+ 个文档块，Chroma 完全没问题。如果未来文档量增长到百万级，可以迁移到 Milvus 或 Pinecone。

### Q73：前端的角色过滤用 CSS 而不是 JS，为什么这么设计？

这是为了**零闪烁**。如果用 JS 做角色过滤，页面渲染完后 JS 才开始隐藏不该看的模块，用户会看到一瞬间的闪烁（不该看到的内容出现了再消失）。用 CSS 做过滤，规则在浏览器渲染第一帧之前就生效了，体验丝滑。具体实现是 `html[data-role="operator"] .role-manager { display: none }` 这种选择器。

### Q74：你们有没有用到什么让你觉得特别酷的技术？

Three.js 的 **MediaPipe 手势识别集成**是最酷的。在 3D 球体页面，摄像头捕捉用户手势，握拳=旋转球体，张开手掌=缩放，两根手指=平移。这本来是给展厅设计的功能，但它展示了一个未来方向：产线巡检人员戴着 AR 眼镜，用手势在空中操控数字孪生界面。

### Q75：这个系统从零开始搭建，你觉得最难的是什么？

最难的不是某个具体技术，而是**让统计、机器学习、大模型、业务规则四种不同的技术范式在同一套系统里和谐协作**。统计模型要可解释，ML 模型要面对数据不足的现实，大模型要约束不乱说，业务规则要灵活可配置。把这几样东西串成一个完整的、能跑通的、产出可执行结果的闭环，是最大的工程挑战。

### Q76：如果让你重新做这个项目，你会改变什么技术选型？

第一，后端可能会用 **FastAPI + SQLAlchemy + Alembic** 做数据库版本管理，因为目前 CSV/SQLite 的方案在数据规模增长后会碰到瓶颈；第二，前端可能会引入一个轻量级框架（如 Alpine.js 或 Petite-Vue）来减少 DOM 操作代码量，当前纯 Vanilla JS 在复杂交互页面（如 324KB 的 index.html）维护成本偏高。

有专门一章"技术创新与亮点"，覆盖六个创新点：数据天花板分析、传感器 ROI 路线图、四级降级架构、SHAP 可解释性流水线、RAG 增强 AI 助手、3D 数字孪生。每个创新点都有"解决了什么问题 → 怎么解决的 → 效果如何"的三段式说明。

### Q79：技术文档对非技术人员友好吗？

我们特别注意了这一点。文档采用分层结构：第一章是"产品概述"，用业务语言讲系统能做什么、给谁用、解决什么问题，不涉及任何技术术语；中间章节逐步深入技术细节；每章开头有"本章要点"摘要。评委和管理者读前两章就能理解核心价值，开发者读后续章节能复现系统。

### Q80：你们的代码有注释吗？有没有 README？

有。CLAUDE.md（项目级开发指南）约 130 行，覆盖项目定位、核心约定、关键路径、常用命令。三个 CLAUDE_REF_*.md 参考文档分别覆盖架构/后端/算法，总计约 3 万字的项目文档。Python 代码中有适度的行内注释，关键决策点（如"为什么不用全局阈值"）有说明。

### Q81：如果别的团队想复现你们的项目，文档够用吗？

够用。我们提供了：
- `requirements.txt`（完整依赖清单，含版本号）
- `CLAUDE.md`（开发环境搭建和启动命令）
- `CLAUDE_REF_*.md`（架构和算法详解）
- `原始数据集/`（4 个脱敏数据文件）
- `.env.example`（环境变量模板）

从零开始到系统跑起来，快的话 15 分钟，慢的话 30 分钟（主要是下载依赖的时间）。

### Q82：文档中是否包含了测试部分？

包含。`CLAUDE.md` 中有完整的测试命令（pytest 单元测试 + 独立报告流测试），`CLAUDE_REF_算法.md` 中有 7 种算法的对比验证结果，`benchmark_algorithms.py` 可在真实数据上重现全部实验结果。

### Q83：你们的文档用什么格式？是否方便评审？

成果文档是 Word（.docx）格式，方便评委审阅和批注。技术参考文档是 Markdown（.md）格式，方便开发者在编辑器中直接查看。代码仓库中的 CLAUDE.md 系列遵循 Markdown 规范，在 GitHub 上可直接渲染。

### Q84：文档中有没有对比分析市面上的同类方案？

有。在"项目背景"章节对比了三种主流方案：传统定期维修（成本高、非必要维修多）、基于规则的告警系统（阈值固定、误报率高）、通用 AI 平台（不了解工业场景、无法给出可执行方案），然后引出我们的差异化定位 — "工业级、全闭环、诚实 AI"。

### Q85：你们的文档是谁写的？有没有用 AI 辅助？

文档由团队成员共同撰写，AI（本项目中使用的 DeepSeek 大模型）辅助进行文字润色和格式整理。核心的技术内容、架构图、数据分析结果全部是我们自己产出的，AI 只负责"翻译成通顺的文字"，不负责"编造内容"。

### Q86：文档提交前有没有做过审核？

有内部交叉审核流程：技术内容由开发者自检（代码和文档是否一致），文字表达由非技术成员审读（是否能看懂），格式排版由专门人员统一（字体、标题层级、图表编号）。确保最终提交的文档专业且易读。

## 七、文档编制能力 — 视频制作（10%）~4问

### Q87：演示视频大概多长？涵盖了哪些内容？

演示视频约 8-10 分钟，按"问题 → 方案 → 演示 → 总结"四段式结构：前 30 秒抛出痛点（工厂运维的困境），接着 2 分钟介绍系统方案和架构，核心 5 分钟做功能演示（从角色选择 → 仪表盘 → AI 助手 → 故障注入 → 工单追踪），最后 1 分钟总结价值。重点突出可操作性和视觉效果。

### Q88：视频里最想让评委记住的三个画面是什么？

第一，**10×10 设备健康矩阵** — 100 个色块随着设备状态动态变化，一眼看出工厂全局健康状态；第二，**AI 助手对话** — 用自然语言问"CNC_023 最近怎么样"，AI 秒回数据+诊断+建议，展示智能化交互；第三，**故障注入 6 步动画** — 从注入故障到生成工单到修复验证，完整闭环，展示系统集成能力。

### Q89：视频有没有配音？有没有字幕？

有真人配音和字幕。配音由团队成员录制，讲述节奏平稳、表达清晰。字幕为简体中文，与配音同步。关键术语和数字（如"83.9% 告警精度"、"0.537 AUC 上限"）在画面上有高亮文字标注，方便评委边看边理解。

### Q90：视频制作过程中，你们特别注意了什么？

特别注意**让非技术人员也能看懂**。避免堆砌技术术语，每个功能演示前先用一句"大白话"说明这个功能解决了什么实际问题。比如演示 SHAP 时不是讲 Shapley 值的数学原理，而是说"这个功能告诉维修师傅，你的设备到底哪里出了问题，别瞎修"。

## 八、路演表现能力 — 路演文稿（15%）~5问

### Q91：路演 PPT 的结构是怎么设计的？

PPT 遵循经典的"黄金圈"结构：**Why**（为什么做 — 工业运维的痛点和数据真相）→ **How**（怎么做 — 系统架构和技术路线）→ **What**（做出了什么 — 功能演示和量化效果）→ **So What**（意味着什么 — 商业价值和未来愿景）。总共约 25-30 页，节奏紧凑，每页一个核心信息点。

### Q92：PPT 的设计风格是什么样的？

深色科技风格，主色调为深蓝+青蓝渐变（呼应工业+AI 主题）。大量使用图表和数据可视化而非大段文字，关键数字（83.9%、0.537、723% ROI 等）用大号字体突出展示。系统截图统一加了圆角阴影和品牌色边框。整体风格专业、现代、有视觉冲击力。

### Q93：路演时你们准备怎么开场？

准备用一个**场景化提问**开场："假设你是工厂厂长，半夜三点被电话叫醒 — 一条产线停机了。你想知道：哪台设备坏了？为什么坏？谁来修？什么时候能修好？修一次多少钱？" 这六个问题对应我们系统解决的六个环节，30 秒内让评委进入场景，理解我们的价值。

### Q94：路演过程的节奏怎么控制？

前 3 分钟：建立认知（问题+方案概述，不碰细节）；中间 4 分钟：核心功能演示（选 3-4 个最出彩的功能快速过，每个不超过 1 分钟）；最后 2 分钟：价值升华（量化效果+商业愿景）。严格控制在 10 分钟内，留 2-3 分钟给评委提问。排练时用计时器卡点。

### Q95：如果路演超时了，你们准备砍掉哪部分？

优先砍技术架构细节页（3 页，纯技术讲解），其次是数据天花板分析的详细推导（2 页，保留结论和 ROI 建议）。功能演示不能砍 — 那是评委最能直观感受系统价值的部分。如果时间真的很紧，用一句"技术细节欢迎在 Q&A 环节深入探讨"带过。

## 九、路演表现能力 — 表达与专业度（15%）~5问

### Q96：路演时谁来讲？怎么分工的？

建议由对业务理解最深的成员主讲（串全场），技术负责人在旁边做操作演示（切页面、点功能）。主讲人负责讲故事和控节奏，演示者负责让系统不出 bug。两人提前至少排练 10 次，配合默契到不看对方也知道谁该做什么。

### Q97：准备怎么应对评委的刁钻提问？

三个策略：第一，**提前准备了这份 100 问**，覆盖了所有评分维度，做到心里有底；第二，**遇到不会的不硬答** — "这个问题我们团队在开发过程中也讨论过，目前的做法是 XXX，我们的思考是在 YYY 方向还需要继续探索" 比硬编一个答案要好得多；第三，**把问题往自己熟悉的方向引导** — 如果问到某个具体算法的数学细节，可以说"算法的核心思想是 XXX，具体实现上我们更关注的是它在工业场景中的落地效果"。

### Q98：你们的路演排练过几次？有什么心得？

（建议回答）排练了至少 10 次以上。最大的心得是：**不要背稿，要讲故事**。背稿的状态是"我下一句该说什么了"，讲故事的状态是"我想让评委理解什么"。心态的转变带来了表达的质变。另外，演示环境一定要提前到场调试 — 投影仪亮度、网络连接、屏幕比例，每一个变量都可能导致翻车。

### Q99：你们觉得自己的项目最大的竞争优势是什么？

**"诚实 AI"的差异化定位。** 市场上大部分 AI 项目都在说我用了什么高端模型、准确率多么高。而我们说："现有数据下，AI 预测不准，这是数据证明的。但不代表 AI 没有用 — 我们用它做异常检测、根因诊断、流程自动化，并通过数据天花板分析告诉你怎么投资升级传感器来突破瓶颈。" 这种坦诚+建设性的态度，在 AI 赛道中非常少见，也更容易赢得评委的信任。

### Q100：最后一个问题 — 你们做这个项目最大的收获是什么？

（建议回答）最大的收获是理解了**"技术在真实场景中的边界"**。在学校里我们学的是"用更复杂的模型提升准确率"，这个项目让我们直面了一个残酷的事实：没有高质量的数据，再牛的模型也没用。同时也学会了**如何向非技术人员讲述技术故事** — 你不需要解释 XGBoost 的梯度提升原理，你只需要让工厂厂长看懂"这一列红色色块意味着你今天应该先修这三台设备"。这种"翻译能力"在未来的职业生涯中可能比写代码更重要。
