数据探索与参数分析

figure_documentation_master · 14项统计任务 · 17组可视化
统计分析 ▶

技术目标

量化评估4个传感器参数（电压、电流、温度、转速）各自对9类故障的区分能力，建立安全运行区间。

电压 Youden's J
0.071
温度 Youden's J
0.060
电流 Youden's J
0.025
转速 Youden's J

技术方法

  • 描述统计 + KDE分布拟合 — 建立正常/故障参数分布画像
  • Cohen's d效应量 + KS检验 — 量化参数区分力（14个任务）
  • Youden's J最优阈值搜索 — 找到最佳单参数切分点
  • 99%分位数安全区间 — 非参数方法建立运行边界
  • Pearson相关矩阵（正常 vs 故障） — 检测故障态参数耦合变化
  • 衍生指标工程：视在功率(V×I)、效率(RPM/A)、热负荷(T/A)
  • 热负荷三级风险框架：P90预警 / P95告警 / P99紧急停机

关键发现

  • 所有4参数 Youden's J < 0.08（可用阈值 > 0.30）
  • 转速完全无诊断价值：Cohen's d = -0.002，KS p = 0.458
  • 正常与故障的参数分布高度重叠（约70%故障样本落在正常范围）
  • 4参数在正常态近乎独立（|r| < 0.09），故障态无显著耦合变化
  • 衍生指标（功率、效率）仅增加微小增量信息

核心局限 → 驱动下一阶段

↓
单参数阈值无法区分故障。需要多参数融合 + 每设备单独基线。
→ 进入阶段2：基线分析与确定

基线分析与统计告警系统

BASELINE_DEVELOPMENT_DOC · 4基线模型 · 三级告警体系
统计基线 ▶

技术目标

建立不依赖ML的统计基线告警系统，作为任何后续模型必须超越的最低性能门槛。

83.9%
Z>2.0 精确率
39.4%
Z>2.0 召回率
19.7%
误报率
100%
T² 精确率

技术方法

  • 基线1：每设备复合Z-Score — 电压/电流/温度三者欧氏组合，每台设备独立 μ/σ（设备间差异占61-73%总方差）
  • 基线2：成本风险矩阵 — Risk = P(fault) × UnitCost × DailyOutput，P90/P75自适应阈值
  • 基线3：故障类型分层阈值 — 3组故障签名（高压/热/微弱），不同z权重
  • 基线4：Hotelling T² 多元SPC — χ²控制限(p=3, α=0.01)，作为高置信度确认通道
  • K-Means冷启动策略 — 稀疏数据设备（<6正常样本）借用聚类基线

关键发现

  • 设备间方差占61-73% — 全局阈值完全不可用
  • Z>2.0为推荐运行点：精确率83.9%，约4/5告警正确
  • Z>1.0的F1=79%最高但误报率80.5%不可接受
  • T²精确率100%但召回率仅22% — 用作验证器而非主检测器
  • 约22%故障样本落在z<2.5范围 — 静态基线完全不可见

核心局限 → 驱动下一阶段

↓
静态基线无时序信息（无法检测漂移趋势）。22%故障在正常z-score范围内。仅7小时观测窗口无法建立退化曲线。需要引入时序特征和ML。
→ 进入阶段3：XGBoost监督学习模型

XGBoost 预测性告警模型

model_development_doc · 4组特征工程 · 双时间窗口
监督学习 ▶

技术目标

引入监督学习预测14/28分钟内的故障风险，超越统计基线性能。

0.480
AUC (28min)
0.475
AUC (14min)
58.3%
先验故障率
~0.50
≈随机猜测

技术方法

  • 特征组A（17维）：滑动窗口时序统计量（均值/标准差/斜率/波动率/安全区间违规比）
  • 特征组B（8维）：设备级静态特征（成本/产量/风险值/老化度/聚类标签）
  • 特征组C（4维）：产品质量特征（15/100设备有数据，其余中位数填充+指示器）
  • 特征组D（6维）：当前状态（最近故障类型/步数/原始值）
  • 双独立XGBoost分类器（14min + 28min）+ 概率一致性约束
  • 成本敏感样本权重：weight[i] ∝ UnitCost × DailyOutput

关键发现

  • AUC ≈ 0.48~0.50 — 等价于随机猜测
  • 模型收敛到"总是预测故障"的平凡解
  • P(next=fault | current=normal) = 71.2% ≈ P(next=fault | current=fault) = 72.5%
  • 故障呈现随机分布，时序依赖性极弱
  • 特征重要性验证：z-score特征排名前三，证实基线方法的信号是真实的
  • 模型价值在风险排序（risk ranking）而非逐点分类

核心局限 → 驱动下一阶段

↓
二分类不是正确任务框架。树模型无法建模时序依赖和参数间复杂交互。需要更深层的特征工程和连续目标变量。
→ 进入阶段4：Multi-Task深度学习模型

Multi-Task 深度学习模型 v2

model_development_doc_v2 · 106维特征 · 4维度特征体系 · 3窗口变体
深度学习 ▶

技术目标

用深度神经网络+多任务学习提升预测能力，从二分类转向故障密度回归。

0.589
最佳AUC (10in→5pred)
0.019

0.537

技术方法

  • 共享特征提取器：128→64→32全连接层 + BatchNorm + ReLU + Dropout(0.35)
  • 故障头（主任务）：回归未来5步故障密度（而非二分类）
  • 质量头（辅助任务）：预测平均产品缺陷数（权重0.30，仅15台设备有标签）
  • 4维度特征体系（106维）：
    维度1-趋势（24+特征）：斜率/Kendall τ/加速度/连续变化计数
    维度2-波动率（36+特征）：标准差/CV/IQR/RMS/峰值因子/波形因子
    维度3-状态（26+特征）：Z-score/安全违规/功率指数/参数相关性
    维度4-成本与上下文（19+特征）：成本风险/设备老化/历史故障率
  • 数据增强：高斯噪声 + 随机掩码 + 特征Dropout（20种噪声条件鲁棒性测试）

关键发现

  • AUC从0.50提升至0.59（+18%相对提升），但来自特征工程改善而非深度学习能力
  • 多任务学习完全失败：质量头R²=0，辅助任务未提供有用梯度信号
  • 模型实际预测的是"高历史故障率的高成本设备"（静态属性）而非实时退化
  • 告警输出：100台中仅7台进入WATCH级别，0台WARNING/ALARM
  • AUC 0.59已接近理论上限（0.5+J_max/2=0.537），更复杂模型不会显著提升

核心局限 → 驱动下一阶段

↓
硬件感知是绑定性约束。纯ML路线已触碰数据的信息理论上限。需要转型：放弃"ML=主告警引擎"，采用决策融合架构。
→ 进入阶段5：系统级决策融合 + AI Copilot

系统级决策融合 + Industrial AI Copilot

predictive_maintenance_system_design · agent-mcp架构 · web-dashboard gateway
决策融合 ▶

技术目标

构建工业级决策融合引擎 + AI Copilot自然语言接口。放弃纯ML路线，拥抱统计+成本+ML+趋势的多信号融合架构。

技术方法 — 决策引擎

  • 4层决策架构：Layer1 多信号融合 → Layer2 异常诊断（4模式）→ Layer3 决策分级（6动作）→ Layer4 维护工单输出
  • 融合权重：统计异常 0.40 + ML密度 0.25 + 成本风险 0.25 + 趋势漂移 0.10
  • DAG流水线：数据准备 → (统计推理 ∥ ML推理) → 诊断 → 决策工单
  • 降级路径：ML不可用时自动切换至纯统计路径（子进程检测依赖可用性）
  • 可预测性上限分析：5维度证据链证明数据瓶颈（单参数无区分力/耦合不变/非渐进故障/模型收敛平凡/传感器缺口）
  • 传感器升级路线图：振动加速度计(P0) → 电流特征分析(P0) → 延长监测周期(P0)

技术方法 — AI Copilot

  • MCP Tool Ecosystem：7个MCP工具（FastMCP + JSON-RPC over stdio）+ 8个知识资源 + 3个Prompt模板
  • FastAPI Gateway（端口8765）：DeepSeek API + 9个OpenAI兼容Tool Definitions
  • 3-阶段对话循环：Phase1 流式LLM调用 → Phase2 工具执行（含chart_data检测）→ Phase3 带工具结果的跟进回复
  • SSE流式传输：text_delta / tool_call / tool_result / chart / error / done 六种事件
  • ECharts可视化集成：工具返回ECharts-ready chart_data → 前端自动渲染折线图/多线对比图/异常标记点/阈值线
  • 系统提示词编码：完整领域知识（9类故障映射、决策权重、传感器局限、问题路由规则）

关键成果

  • 9个MCP工具覆盖完整运维流程（设备状态→告警列表→故障历史→传感器趋势→设备对比→根因分析→工单历史→可预测性解释→全流水线）
  • 自然语言 → 工具调用 → 真实数据 → ECharts图表 → AI分析，全链路闭环
  • 统计基线(P=83.9%)确保了即使ML信号极弱时系统依然可用
  • 5维度证据链系统性证明了"为什么无法预测"（非盲目调参）
  • 工业深色主题Web Dashboard + AI Chat界面
