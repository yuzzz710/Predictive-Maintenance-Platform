# 虚拟数据集生成方案 v1.0

> **目标**：基于原始4个CSV的真实统计特征，生成覆盖100台设备、每台300条日志、全部设备具备产品数据的虚拟数据集，用于模拟项目投入真实工厂后的运行效果。
>
> **原则**：所有生成逻辑基于原始数据的统计分布，不凭空编造模式。输出文件格式必须与原始文件完全一致（列名、列序、编码、分隔符）。

## 二、输出：虚拟数据集规格

### 2.1 MACHINE_LOG — 从30条/台扩展到300条/台

| 属性 | 原值 | 新值 |
|------|------|------|
| 设备数 | 100 | 100 |
| 每台记录数 | 29~30 | **300** |
| 总行数 | 2,999 | **30,000** |
| 时间跨度 | 1天（7小时） | **10天**（每天30条，0:00~6:57） |
| 日期范围 | 2025/5/18 | **2025/5/18 ~ 2025/5/27** |

**生成规则：**

1. **时间轴**：每天重复相同的30个时间戳（0:00, 0:14, 0:28...6:57），日期递增。每天10天 × 每天30条 × 100台 = 30,000条。

2. **正常状态传感器值**：对每台设备i，从其原始正常(Type=0)样本中估算μ_i和σ_i（4个参数各一组）。每天的正常数据：
   ```
   value = μ_i + ε,  ε ~ N(0, σ_i²)
   ```
   加入轻微的日内漂移（模拟温升）：Temperature和Amperage随时间戳递增略有上升（每天前5条+0.5%/条，之后平稳）。

3. **故障注入**：不是每天都故障。设计一个**设备健康日历**：
   - 10天中，每台设备有3~8天为完全正常（全部Type=0）
   - 故障日：按原始故障类型分布比例（Type1~9对应比例）随机选择故障类型
   - 同一设备在不同天的故障类型可以不同（模拟真实工厂的多发故障模式）
   - 故障时的参数偏移遵循原始数据的偏移模式（见1.1表）

4. **故障渐进性**（增强真实感）：
   - 故障前2~3条记录（约30-45分钟前）参数开始缓慢漂移向故障值
   - 例如：CNC_036某天Type3故障，则前3条Temperature从正常值线性上升到故障值

5. **跨设备相关性保留**：
   - 保留原始数据中 Voltage-Ampetage 的正相关（ρ≈0.4）
   - 保留 Temperature-Ampetage 的微弱正相关（ρ≈0.2）
   - 保留 Rotor Speed 独立于其他参数的属性

### 2.2 MACHINE_SUMMARY — 保持100行不变

Summary数据本身已覆盖100台，不需要扩展行数。但为确保虚拟数据集的自洽性，需要微调：

- **Unit Cost** 分布保持原有右偏特征（1~114，中位数≈3）
- **Units Produced Per day** 按原有分布随机分配给新增的85台设备
- **Month of Manufacture** 在 202001~202303 范围内均匀分布
- **服务日期** 以 LOG 数据的最后一天（2025/5/27）为参考点，距上次保养3~107天，距下次保养-10~+131天

Summary数据**保持不变**（直接用原始的100行），因为原始100台设备的信息已经完整。

### 2.3 PRODUCT_ASSEMBLY — 从15台扩展到100台

| 属性 | 原值 | 新值 |
|------|------|------|
| 设备数 | 15 | **100** |
| 每台产品数 | 9 | **9** |
| 总行数 | 135 | **900** |

**生成规则：**

1. **产品编号体系**（沿用原始3级层次）：
   - 100台设备 → 100个CASE（CA1~CA100）
   - 每个CASE含2~3个CIRCUIT（CRxx）
   - 每个CIRCUIT含2~3个CAPACITOR（Cxxx）
   - 每台设备测试3个CAPACITOR + 3个CIRCUIT + 3个CASE = 9个产品

2. **FAILED_TESTS 生成**：
   - 基于该设备的健康状态（从LOG数据统计的故障率）决定产品质量
   - 故障率 < 20% 的设备：FAILED_TESTS以0为主（均值0.3）
   - 故障率 20-50% 的设备：FAILED_TESTS中等（均值1.5）
   - 故障率 > 50% 的设备：FAILED_TESTS较高（均值3.0）
   - 具体分布对齐原始数据：0(53%), 1(10%), 2(19%), 3(7%), 4(11%)

3. **测试列**：二元测试列（ANALOG TESTS等）按原始比例随机生成0/1。HIGH_RANGE_VALUE_TESTS和LOW_RANGE_VALUE_TESTS在0~4间分布。

4. **工作站和产线**：CAPACITOR_LOAD / CIRCUIT_LOAD / CASE_LOAD 各占1/3。Line1/Line2/Line3按比例分布。

5. **PARENT_CASE_NO / PARENT_CIRCUIT_NO的NULL逻辑**：
   - CAPACITOR_LOAD：两者都有值
   - CIRCUIT_LOAD：PARENT_CIRCUIT_NO=NULL
   - CASE_LOAD：两者都=NULL

### 2.4 PRODUCT_TESTS — 从15台扩展到100台

| 属性 | 原值 | 新值 |
|------|------|------|
| 设备数 | 15 | **100** |
| 行数 | 420 | **~2,800** |
| 每台记录 | 17~32 | **28**（均值对齐） |

**生成规则：**

1. **PARAMETER分布**（对齐原始9种参数的比例）：
   - Boundary scan tests: 18.6%
   - Discharging capacitors: 18.6%
   - Contact Test: 18.6%
   - Power up UUT: 18.6%
   - Testjet: 6.2%
   - Analog tests: 6.2%
   - Powered analog: 6.2%
   - Shorts testing: 3.6%
   - FrameScan: 3.6%

2. **MEASMT_VALUE 生成**：
   - 原始分布近似对数正态，min=5, max=155, mean=43, std=34
   - 合规值(在[10,100]内)占64%，用截断正态分布生成，mean=45, std=25
   - 超规格值(<10或>100)占36%，用均匀分布生成[1,9]和[101,160]

3. **超规格率与设备健康关联**：
   - 健康设备（故障率<20%）：超规格率5-15%
   - 一般设备（故障率20-50%）：超规格率25-45%
   - 高危设备（故障率>50%）：超规格率50-90%

4. **SPEC_LIMIT**：全部保持 LWR=10, UPR=100（与原始一致）

5. **日期**：2025/5/28 ~ 2025/6/1（LOG数据最后一天之后）

6. **UNIT_OF_MEAS**：
   - farad: 对应CAPACITOR_LOAD的大多数测试（~74%）
   - Voltage: 对应CIRCUIT_LOAD测试（~19%）
   - Pass/Failed: 对应POWER UP, POWERED ANALOG, SHORTS TESTING（~7%）
   - **修复原始数据的typo**：`Falied` → `Failed`

## 三、生成算法伪代码

### 3.1 LOG生成算法

```
FOR each machine i in 1..100:
    # Step 1: 从原始数据估算设备参数
    normal_data_i = 原始LOG中 machine_i 且 Type=0 的记录
    μ_i = {V: mean(normal_data_i.Voltage), A: mean(normal_data_i.Amperage),
           T: mean(normal_data_i.Temperature), R: mean(normal_data_i.Rotor_Speed)}
    σ_i = {V: std(normal_data_i.Voltage),  A: std(normal_data_i.Amperage),
           T: std(normal_data_i.Temperature), R: std(normal_data_i.Rotor_Speed)}

    # Step 2: 生成10天的健康日历
    fault_days_i = 随机选择 3~8 天（每台不同）
    FOR each day in 1..10:
        date = 2025/5/18 + (day-1) 天
        IF day in fault_days_i:
            fault_type = 按原始Type1~9比例随机选择
            fault_start_step = 随机选择 25~29（故障在最后几条发生）
        ELSE:
            fault_type = 0（整日正常）
            fault_start_step = 999（永不触发）

    # Step 3: 生成每天30条记录
    FOR each step in 0..29:
        timestamp = date + timestamps[step]  # 每天相同的时间网格

        IF fault_type > 0 AND step >= fault_start_step:
            # 故障记录
            current_type = fault_type
            # 根据故障组施加参数偏移
            IF fault_type in (4,5):    # High-Voltage
                V = μ_i.V + σ_i.V*ε + δ_V   # δ_V ~ N(+10, 3)
                A = μ_i.A + σ_i.A*ε
                T = μ_i.T + σ_i.T*ε
            ELIF fault_type in (3,6,7,8,9):  # Thermal
                V = μ_i.V + σ_i.V*ε
                A = μ_i.A + σ_i.A*ε
                T = μ_i.T + σ_i.T*ε + δ_T   # δ_T ~ N(+2, 1)
            ELSE:  # Subtle
                V = μ_i.V + σ_i.V*ε
                A = μ_i.A + σ_i.A*ε + δ_A   # δ_A ~ N(+0.5, 0.3)
                T = μ_i.T + σ_i.T*ε
        ELIF fault_type > 0 AND step >= fault_start_step - 3:
            # 故障前渐进漂移（3步线性过渡）
            progress = (step - (fault_start_step - 3)) / 3
            # 用progress比例混合正常值和故障值
        ELSE:
            # 正常记录
            current_type = 0
            V = μ_i.V + σ_i.V * N(0,1)
            A = μ_i.A + σ_i.A * N(0,1)
            T = μ_i.T + σ_i.T * N(0,1)

        R = μ_i.R + σ_i.R * N(0,1)  # 转速始终独立
        施加日内温升效应
        施加V-A正相关约束
```

### 3.2 ASSEMBLY生成算法

```
FOR each machine i in 1..100:
    fault_rate_i = 从LOG数据统计 machine_i 的总体故障率

    # 生成9个产品
    products = []
    case_id = f"CA{i}"
    FOR circuit_j in 1..3:
        circuit_id = f"CR{i*10 + circuit_j}"
        FOR capacitor_k in 1..3:
            capacitor_id = f"C{i*100 + circuit_j*10 + capacitor_k}"
            # 按工作站类型生成
            products.append({type: CAPACITOR_LOAD, serial: capacitor_id, parent_circuit: circuit_id, parent_case: case_id})
        products.append({type: CIRCUIT_LOAD, serial: circuit_id, parent_case: case_id})
    products.append({type: CASE_LOAD, serial: case_id})

    FOR each product in products:
        # 根据设备故障率决定FAILED_TESTS
        base_p = fault_rate_i  # 设备故障率映射为产品缺陷概率
        failed = 从多项分布采样(基于base_p调整后的分布)
        二元测试列 = Bernoulli(各自的基础概率)
        HIGH_RANGE = Poisson(1.5 + base_p * 3)
        LOW_RANGE = Poisson(1.5 + base_p * 3)
```

### 3.3 TESTS生成算法

```
FOR each machine i in 1..100:
    fault_rate_i = 从LOG统计的故障率
    oos_rate_i = 映射: fault_rate<20%→5-15%, 20-50%→25-45%, >50%→50-90%

    # 为每台生成约28条测试记录
    FOR each serial (产品的SERIAL NO，与ASSEMBLY对应):
        # 每个serial测试3-4个参数
        test_params = 从9种参数中随机选3-4个（按原始比例加权）
        FOR each param:
            IF random() < oos_rate_i:
                measmt = 超规格值：Uniform(1,9) 或 Uniform(101,160)
            ELSE:
                measmt = 截断正态(mean=45, std=25, lower=10, upper=100)
            LWR = 10, UPR = 100  # 始终不变
```

## 四、数据一致性约束（必须满足）

1. **LOG内部一致性**：同一时间戳下所有100台设备都有一条记录。每天恰好3000条（100台×30步）。CNC_100不再缺失最后一条。

2. **ASSEMBLY ↔ TESTS 设备对应**：ASSEMBLY中出现的MACHINE必须在TESTS中出现，反之亦然。两台文件使用完全相同的100台设备。

3. **ASSEMBLY ↔ TESTS 产品对应**：TESTS中的SERIAL NO必须是ASSEMBLY中SERIAL NO的子集（每个serial在TESTS中出现2-4次，因为每个serial测多个参数）。

4. **故障率关联链**：LOG中故障率高的设备 → ASSEMBLY中FAILED_TESTS高 → TESTS中超规格率高。这是真实世界的因果链：设备不稳定 → 产品质量差。

5. **SUMMARY ↔ LOG 日期一致性**：SUMMARY中Last Service Date / Next Service Date / Last Repair Date以LOG最后一天（2025/5/27）为参考，使 `days_since_service` 等计算值落在合理范围（3~107天）。
