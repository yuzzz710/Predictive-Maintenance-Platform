## 1. v2 设计动机与改进方向

### 1.1 v1 的经验教训

v1（XGBoost 双阶段模型）在训练后揭示了数据的根本局限：

| v1 发现 | 数值 | 含义 |
|----------|------|------|
| Test AUC | ~0.50 | 等价随机猜测 |
| 模型行为 | Always Predict Fault | 收敛到"永远预测故障" |
| Recall | 100% | 无区分能力 |
| Precision | 72.5% | 等于故障先验概率 |

v1 不是代码错误——基线分析已证实：4 个监控参数（Voltage, Amperage, Temperature, Rotor Speed）在正常/故障状态间的可区分性极低（单参数 Youden's J < 0.08，70% 故障样本的参数值在正常范围内）。

### 1.2 v2 的改进策略

面对"参数区分度不够"这一根本约束，v2 从以下方向寻求突破：

| 方向 | v1 做法 | v2 改进 | 预期效果 |
|------|---------|---------|----------|
| 预测窗口 | 14min / 28min 单步 | 5 步预测窗口（~70min） | 捕捉更长期的退化趋势 |
| 特征工程 | 基础统计特征 | 四维特征体系（趋势/波动/状态/成本） | 提取更深层次的信号 |
| 辅助任务 | 无 | 产品质量预测（多任务学习） | 利用额外数据源增强表征 |
| 数据增强 | 无 | 噪声+随机缺失+时间扰动 | 小样本下提升鲁棒性 |
| 告警逻辑 | 单点概率阈值 | 连续确认+多信号融合 | 消除抖动，工业可用 |
| 模型架构 | XGBoost | Multi-Task Neural Network | 共享表征，联合优化 |

## 2. 模型架构设计

### 2.1 总体架构

```
                    ┌─────────────────────────────┐
                    │    106-Dim Feature Vector     │
                    │  Trend │ Volatility │ State │ Cost │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │     Shared Feature Extractor  │
                    │  Dense(128) + BN + ReLU + DO  │
                    │  Dense(64)  + BN + ReLU + DO  │
                    │  Dense(32)  + BN + ReLU + DO  │
                    └──────┬───────────────┬──────┘
                           │               │
              ┌────────────▼──┐   ┌────────▼──────────┐
              │  Fault Head   │   │  Quality Head      │
              │  Dense(16)+ReLU│   │  Dense(16)+ReLU    │
              │  Dense(1)→σ    │   │  Dense(1)          │
              │  MSE Loss      │   │  MSE Loss          │
              └──────┬─────────┘   └────────┬───────────┘
                     │                      │
              Fault Density           Quality Score
              y ∈ [0, 1]              y ∈ R
```

### 2.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 共享层维度 | [128, 64, 32] | 三层递减，逐步压缩到紧凑表征 |
| Dropout | 0.35（共享层）/ 0.25（任务头） | 强正则化对抗小样本过拟合 |
| BatchNorm | 每层激活前 | 稳定训练，加速收敛 |
| 激活函数 | ReLU | 标准选择，梯度稳定 |
| 故障头输出 | 1 维 + Sigmoid → [0, 1] | 直接输出故障密度预测 |
| 质量头输出 | 1 维（无激活） | 回归任务，输出范围不限 |
| 总参数量 | ~25,570 | 轻量级，适合 CPU 推理 |

## 3. 四维特征工程体系

### 3.1 维度一：趋势特征（Trend）

**目标**：回答"参数正在朝哪个方向变化？"

| 特征类别 | 特征名 | 数量 | 物理含义 |
|----------|--------|------|----------|
| 线性趋势 | `slope`, `trend_r2`, `kendall_tau` | 3×3=9 | 参数是否在持续漂移 |
| 加速度 | `accel_mean`, `accel_abs_max` | 2×3=6 | 漂移是否在加速 |
| 方向持续性 | `consecutive_up/down`, `reversals` | 3×3=9 | 单调性 vs 振荡 |

### 3.2 维度二：波动特征（Volatility）

**目标**：回答"参数有多不稳定？"

| 特征类别 | 特征名 | 数量 | 物理含义 |
|----------|--------|------|----------|
| 离散度 | `std`, `cv`, `range`, `iqr`, `mad` | 5×3=15 | 基本波动量 |
| 工业指标 | `rms`, `crest_factor`, `form_factor` | 3×3=9 | 电气/振动监测标准指标 |
| 差分统计 | `diff_std`, `diff_max`, `diff_range` | 3×3=9 | 逐步变化幅度 |
| 交叉比值 | `cv_ratio_v_a`, `cv_ratio_v_t`, `cv_ratio_a_t` | 3 | 参数间相对波动 |

### 3.3 维度三：状态特征（State）

**目标**：回答"设备当前处于什么工况？"

| 特征类别 | 特征名 | 数量 | 物理含义 |
|----------|--------|------|----------|
| 故障历史 | `current_fault_type`, `window_fault_ratio`, `fault_diversity` | 4 | 窗口内故障密度和类型 |
| Z-Score | `z_v/a/t_last`, `z_comp_last/mean/max/std/trend` | 8 | 偏离正常基线的程度 |
| 安全范围 | `v/a/t_violation_count/severity` | 6 | 超出安全阈值的次数和程度 |
| 复合指标 | `power_mean/trend`, `thermal_mean/max/over_p95` | 5 | 功率、热应力 |
| 参数耦合 | `corr_va/vt/at` | 3 | 窗口内参数间相关性的变化 |

### 3.4 维度四：成本与上下文特征（Cost & Context）

**目标**：回答"这台设备有多重要？历史维护情况如何？"

| 特征类别 | 特征名 | 数量 | 物理含义 |
|----------|--------|------|----------|
| 经济要素 | `unit_cost`, `daily_output`, `cost_at_risk` | 3 | 故障的经济后果 |
| 服役状态 | `machine_age_months`, `days_since_service/repair` | 4 | 设备老化与维护时差 |
| 历史故障 | `historical_fault_rate`, `steps_since_last_fault` | 2 | 故障频率和间隔 |
| 产品质量（仅15台） | `prod_*` 系列特征 | ~10 | 产品测试通过率、偏差等 |

## 4. 多任务学习设计

### 4.1 任务定义

| 项目 | 主任务（Fault） | 辅助任务（Quality） |
|------|----------------|---------------------|
| 预测目标 | 未来 5 步的故障密度 y ∈ [0, 1] | 平均单台产品缺陷数 |
| 损失函数 | MSE Loss | MSE Loss |
| 损失权重 | 1.0 | 0.30 |
| 数据覆盖 | 100 台设备 | 仅 15 台设备 |
| 缺失处理 | — | 对无质量数据的 85 台，mask 辅助损失 |

### 4.2 为什么选择回归而非分类

v1 使用二分类标签（下一步是否有故障），但数据显示：

```
P(未来5步至少一个故障) = 1 - P(全正常) = 1 - 0.28^5 ≈ 0.998
```

几乎所有窗口都是正类，二分类模型无法学习区分。改用**故障密度回归**后：

- 标签分布：均值 0.72-0.76，范围 [0, 1]
- 可以学习预测"故障的严重程度"而非简单的"有/无"
- 对于运维，知道"未来 70 分钟内有 80% 的时间会出故障"比"会出故障"更有用

### 4.3 产品质量特征构建

从产品测试数据中提取每台设备的聚合特征：

```python
build_product_features():
    # 组装维度
    avg_failed_tests, max_failed_tests, defect_rate
    # 测试维度
    test_pass_rate, param_deviation_mean, param_deviation_max
```

仅 15 台设备（CNC_001 到 CNC_015）有产品质量数据。

## 5. 数据增强与鲁棒性策略

### 5.1 训练时增强

```python
DataAugmenter.augment(X):
    1. 高斯噪声：噪声强度 = random(0.02, 0.05) × 特征标准差
    2. 随机掩码：20% 概率随机遮挡 1-3 个特征（模拟传感器故障）
    3. 特征 Dropout：5% 概率随机置零单个特征值
```

### 5.2 鲁棒性评估

在 4 个噪声水平 × 5 个掩码概率 = 20 种条件下各测试 10 次：

| 条件 | MSE 变化 | AUC 变化 | 结论 |
|------|---------|---------|------|
| 无扰动 | 0.0361 | 0.589 | 基线 |
| 噪声 5%, 掩码 0% | 0.0361 | 0.587 | 几乎无影响 |
| 噪声 5%, 掩码 20% | 0.0363 | 0.585 | 极小影响 |
| 噪声 15%, 掩码 30% | 0.0367 | 0.553 | 轻微退化，可接受 |

模型在强扰动下表现稳定，说明特征间存在冗余，单一特征的缺失不会导致预测崩溃。

## 6. 训练策略与防泄露设计

### 6.1 数据集划分

采用**设备级划分**（Machine-Level Split），杜绝同一设备的数据跨越训练/验证/测试：

```
100 台设备 → 随机打乱 (seed=42)
  ├── 训练：60 台设备的所有窗口
  ├── 验证：20 台设备的所有窗口
  └── 测试：20 台设备的所有窗口
```

**为什么不是时间划分？** 仅 30 个时间步，对于需要 15-20 步的窗口，按时间划分会导致训练集为 0。设备级划分同样杜绝了数据泄露（同一设备的参数特征不会在训练和测试中同时出现）。

### 6.2 多窗口变体

| 变体 | 输入步数 | 预测步数 | 总跨度 | 训练样本 |
|------|---------|---------|--------|---------|
| 15in_5pred | 15 (~3.5h) | 5 (~70min) | 20 (~4.7h) | 660 |
| 10in_10pred | 10 (~2.3h) | 10 (~2.3h) | 20 (~4.7h) | 660 |
| 10in_5pred | 10 (~2.3h) | 5 (~70min) | 15 (~3.5h) | 960 |

### 6.3 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 优化器 | AdamW | 权重衰减 1e-4 |
| 学习率 | 0.001 | 初始学习率 |
| LR 调度 | ReduceLROnPlateau | patience=25, factor=0.5 |
| Batch Size | 64 | 带加权采样 |
| 最大 Epoch | 500 | 早停 patience=60 |
| 类别平衡 | WeightedRandomSampler | 按标签值加权 |
| 梯度裁剪 | max_norm=1.0 | 防止梯度爆炸 |
| 早停准则 | Val Loss (Fault MSE + 0.3 × Quality MSE) | mode=min |

### 6.4 基线计算防泄露

每台设备的正常态基线（μ, σ）仅从训练集正常样本（time_step < 20）中计算，确保验证/测试集的特征提取不会泄露未来信息。

## 7. 告警逻辑设计

### 7.1 多信号融合

```python
alert_score = 0.40 × ml_density    # 模型预测的故障密度
            + 0.35 × stat_anomaly   # Z-Score 统计异常 (0-1)
            + 0.25 × cost_risk      # 成本风险归一化 (0-1)
```

### 7.2 连续确认机制

| 告警级别 | 升级所需连续窗口 | 降级所需连续窗口 | 分数阈值 |
|----------|-----------------|-----------------|---------|
| ALARM | 3 | 3 | ≥ 0.70 |
| WARNING | 2 | 2 | ≥ 0.50 |
| WATCH | 1 | 2 | ≥ 0.30 |
| NORMAL | — | — | < 0.30 |

设计原则：
- **防抖**：升级需要连续确认，避免单次噪声触发误报
- **滞后**：降级也需要连续确认，避免告警在阈值附近震荡
- **分级**：不同级别对应不同的运维响应策略

## 8. 实验结果与分析

### 8.1 故障密度预测（主任务）

| 变体 | MSE ↓ | R2 ↑ | MAE ↓ | BinAUC ↑ |
|------|-------|------|-------|----------|
| 15in_5pred | 0.0368 | 0.0114 | 0.1551 | 0.5469 |
| 10in_10pred | 0.0204 | -0.1737 | 0.1167 | 0.2166 |
| **10in_5pred** | **0.0361** | **0.0188** | **0.1594** | **0.5888** |

**解读**：
- **R2 ≈ 0 或负数**：模型的预测能力不优于直接预测均值（"永远预测 0.73"的基线）
- **BinAUC 0.55-0.59**：略优于随机猜测（0.5），但远未达到工业可用水平（通常要求 > 0.80）
- **MSE 0.02-0.04**：在 [0,1] 区间上看起来不算大，但真实密度的标准差仅 ~0.20，所以 MSE 0.04 意味着预测几乎等于均值
- **10in_10pred 负 R2**：说明 10 步预测 10 步的窗口设置过宽，模型学习到了错误模式

### 8.2 产品质量预测（辅助任务）

| 变体 | Quality MSE | Quality R2 | Quality AUC |
|------|------------|-----------|-------------|
| 15in_5pred | 0.1341 | 0.0000 | 0.5000 |
| 10in_10pred | 0.3718 | 0.0000 | 0.5000 |
| 10in_5pred | 0.2557 | 0.0000 | 0.5000 |

**解读**：辅助任务在所有变体上 R2 = 0，AUC = 0.5（随机），意味着：
1. 产品质量特征对故障预测没有提供有用的梯度信号
2. 多任务学习的共享表征未能从质量任务中获益
3. 仅 15 台设备有质量数据（占训练集 25%），样本量不足以让辅助任务收敛

### 8.3 告警输出分布

```
ALARM:   0 machines
WARNING: 0 machines
WATCH:   7 machines
NORMAL:  93 machines
```

当前告警配置偏保守。7 台 WATCH 级别的设备主要是高成本风险设备（CNC_085, CNC_025, CNC_048 等），告警分数由 cost_risk 主导而非 ML 预测驱动。

### 8.4 鲁棒性评估

模型在噪声和缺失数据下表现极为稳定——MSE 标准差 < 0.001，AUC 标准差 0.03-0.05。这不是因为模型好，而是因为模型已经退化为"均值预测器"，均值对扰动不敏感。

## 10. 多任务学习有效性分析

### 10.1 假设与验证

v2 的核心假设：**产品质量预测作为辅助任务能增强共享层的表征能力，从而提升故障预测的 AUC > 5%。**

### 10.2 验证结果

| 指标 | 预期 | 实际 | 达标？ |
|------|------|------|--------|
| Quality R2 | > 0 | 0.00 | 否 |
| Quality AUC | > 0.60 | 0.50 | 否 |
| Fault AUC 提升 | > 5% | ~8%（vs v1） | 是，但来自特征工程 |
| 共享表征质量 | 有区分度 | 退化为均值 | 否 |

### 10.3 失败原因

1. **数据覆盖不足**：仅 15% 设备有质量数据，辅助梯度稀疏
2. **标签噪声**：`avg_failed_tests` 是一个高度聚合的统计量（每台设备一个值），不是时序对齐的标签
3. **信噪比过低**：监控参数本身的信息量已经极低，加入弱相关任务无法"创造"额外信息
4. **表征坍缩**：在信息不足时，共享层的最优解是输出常数（故障先验概率），两个任务头都无法提供有意义的梯度方向

### 10.4 什么情况下多任务学习会有效

多任务学习在以下条件下有效：
- 辅助任务有充足且时序对齐的标签（本数据集：每台设备仅一个静态质量标签）
- 主任务本身有足够信噪比（本数据集：4 参数区分度 < 0.08）
- 两个任务共享底层物理机理（本数据集：监控参数与产品缺陷 r = 0.075，几乎无关）

本数据集不满足以上任何条件。

---

## 11. 数据局限性根本原因分析

### 11.1 问题根源

经过 v1 和 v2 两轮建模，可以确认：**这不是模型或方法的问题，而是数据信息量的问题。**

```
故障检测的信息瓶颈：

  4 个监控参数 (Voltage, Amperage, Temperature, Rotor Speed)
       │
       ├── 各参数在正常/故障间 Cohen's d < 0.12
       │       → 效应量"极小"
       │
       ├── 70% 故障样本的参数值在正常范围内 (基线 Task 9)
       │       → 大部分故障不以参数偏移的形式表现
       │
       ├── 参数间几乎独立 (|r| < 0.09)
       │       → 没有可利用的交互效应
       │
       ├── Rotor Speed 在正常/故障间 KS p = 0.46
       │       → 完全无诊断价值
       │
       └── 产品测试与监控参数 r = 0.075
               → 质量信号无法通过监控参数传递

  结论：以这 4 个参数为基础的任何模型（统计/ML/DL），
        其预测能力的理论上限受限于参数本身的信息量。
```

### 11.2 Youden's J 指数的工程解释

Youden's J = Sensitivity + Specificity - 1，范围 [-1, 1]，0 表示完全无区分能力。

| 参数 | Best Youden's J | 工程含义 |
|------|----------------|---------|
| Voltage | 0.075 | 用电压筛故障，每 100 次告警约 54 次正确 |
| Amperage | 0.042 | 电流的区分力弱于电压 |
| Temperature | 0.038 | 温度的区分力最弱 |
| Rotor Speed | 0.014 | 几乎无用 |

即使在最佳阈值下，电压也只能获得 Sensitivity + Specificity ≈ 1.075，远低于可用标准（通常要求 > 1.5，即 Sensitivity 和 Specificity 之和至少 2.5）。

### 11.3 模型能达到的理论上限

假设我们能完美提取 4 个参数中的所有信息：

```
信息量上限 ≈ max(Youden's J) = 0.075

这等价于：
  - 最佳可能的 AUC ≈ 0.5 + J/2 ≈ 0.54
  - 最佳可能的 F1 ≈ 基于先验概率加权 ≈ 0.55-0.60

我们的 v2 模型 AUC = 0.59，已经非常接近理论边界。
```

## 12. 工程部署建议

### 12.1 当前模型是否可用？

**不推荐部署为独立告警系统。** 原因：
- BinAUC 仅 0.59，误报和漏报率都不可接受
- 主要预测信号来自成本特征（静态），非参数退化（动态）
- 告警本质上是"高成本设备 + 高历史故障率"的静态排序，不是预测性维护

### 12.2 如果必须部署，如何最大化效用？

| 策略 | 具体做法 | 效果 |
|------|---------|------|
| 降级使用 | 将模型输出作为"关注列表"而非"告警" | 7 台 WATCH 设备可安排优先巡检 |
| 结合基线 | 配合 z-score 基线（P=84%, FPR=20%） | 统计基线 + ML 双重确认 |
| 成本优先 | 按 cost_at_risk 排序派单 | 经济效用最大化 |
| 人工审核 | 模型输出需运维确认 | 防止自动化误判 |

### 12.3 数据采集建议（如需改进模型）

| 优先级 | 新增数据 | 预期提升 | 理由 |
|--------|---------|---------|------|
| P0 | 振动/声发射传感器 | AUC +0.15-0.25 | 振动是最敏感的机械故障前兆 |
| P0 | 更长监控周期（>3个月） | AUC +0.10 | 建立退化趋势和寿命曲线 |
| P1 | 环境参数（湿度、粉尘） | AUC +0.05-0.10 | 环境加速老化效应 |
| P1 | 操作员/班次信息 | AUC +0.03-0.05 | 操作差异导致的质量偏差 |
| P2 | 维修工单文本 | — | NLP 提取故障模式和根因 |

### 12.4 替代方案

在现有数据约束下，最有效的"预测性维护"策略是：

1. **基于基线 1 的统计告警**（z > 2.0，P=84%，FPR=20%）——这是当前数据下最有区分力的方法
2. **基于基线 2 的成本风险排序**——确定有限维护资源的最优分配
3. **ML 模型作为辅助参考**——与其他信号交叉验证，不单独决策
