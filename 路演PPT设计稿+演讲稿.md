## 第 2 页｜背景与痛点

**1. 本页核心结论：**
> 100台CNC机床的被动维护模式，正在吃掉工厂的利润和产能。

**2. 页面文字：**
- 场景规模：100 台 CNC 数控机床，4 维传感器（电压 / 电流 / 温度 / 转速）
- 三大痛点：
  - 被动维修：坏了才修，非计划停机损失高达产能 15%~30%
  - 人工巡检：依赖经验判断，不同班组标准不一，漏检率难以控制
  - 策略缺失：成本效率 / 生产效率 / 质量优先——没有量化工具支撑决策
- 核心命题：如何用数据驱动的预测性维护，把"被动抢修"变为"主动防御"？

**3. 图片占位设计：**
- [右侧 1/3 区]：工厂产线实拍图或 CNC 机床特写（暗调处理，不抢文字焦点）
- [底部]：三个痛点横向排列的图标卡片（故障图标/巡检图标/天平图标）

**4. 版式建议：**
左右分栏——左 2/3 放文字（痛点竖向排开），右 1/3 放场景图。三个痛点各配小图标。

**5. 演讲稿（约 100 字）：**
> 我们的场景很具体：100台CNC数控机床，每台采集电压、电流、温度、转速四个参数。这个规模在制造业很典型。但典型的也是落后的——现在普遍是坏了才修，非计划停机占产能损失的15%到30%。靠老师傅的经验巡检，标准不统一、漏检难控制。成本、效率、质量这三个目标怎么权衡？没有工具能算清楚。我们想做的是把"被动抢修"变成"主动防御"——这就是预测性维护的核心价值。

**6. 关键数字/关键词：**
100 台 CNC · 4 维传感器 · 15%-30% 产能损失 · 被动→主动

## 第 3 页｜关键发现：数据瓶颈

**1. 本页核心结论：**
> 仅靠4个传感器做纯ML预测，信息量不够。这不是模型的问题，是输入的问题。（★ 全篇转折页）

**2. 页面文字：**
- 结论一：4 参数的 Youden's J 指数最高仅 0.075，理论 AUC 上限 ≈ 0.537
- 结论二：7 种主流算法（RF / XGBoost / LightGBM / SVM / LR / KNN / MLP）实测 AUC 全部落在 0.517–0.589，接近随机猜测
- 结论三：设备间方差占 61%–73%，全局阈值无效——每台设备必须建立独立基线
- 我们的判断：**不堆模型，换路线。**

**3. 图片占位设计：**
- [左侧 1/2]：Youden's J 分布图或 KDE 叠加图（4参数×200点分布曲线，突出低分离度）
- [右侧 1/2]：7 算法 AUC 对比柱状图，在 0.5 附近画一条醒目的红色虚线标注"随机基线"
- [图下方]：一行大字标注" → 转向多信号融合 + 统计基线方案"

**4. 版式建议：**
左右分栏——左放 KDE 分布图，右放柱状图。本页以图为主，文字只做结论标注。正中间用箭头或分隔线强化"转折感"。

**5. 演讲稿（约 130 字）：**
> 这是整个项目最关键的一页。在正式设计方案之前，我们先做了一件事：验证数据到底够不够。我们对4个传感器参数做了完整的信息量评估，用Youden's J指数衡量它们的故障判别能力——结果最高只有0.075。什么意思？接近零。理论AUC上限算出来是0.537，几乎等于抛硬币。我们不信邪，跑了7种主流算法交叉验证——随机森林、XGBoost、LightGBM、SVM、逻辑回归、KNN、MLP——实测AUC全部在0.52到0.59之间，和理论预测完全吻合。更有意思的是方差分解：设备间差异占61%到73%，说明每台机床的"正常"都不一样。这个结论推导出一个铁律：必须逐设备建基线，必须用多信号融合来补信息缺口。所以我们决定——不堆模型，换技术路线。

**6. 关键数字/关键词：**
Youden's J ≤ 0.075 · AUC 上限 0.537 · 7算法 0.517–0.589 · 设备方差 61–73% · **换路线**

## 第 4 页｜解决方案架构

**1. 本页核心结论：**
> 统计基线发现异常，多信号融合做出决策——这是一套会解释"为什么"的工业系统。

**2. 页面文字：**
- 逐设备 Z-Score 基线 + Hotelling T² 多变量检测（≥3 样本建基线，三层回退策略）
- 四路信号融合：统计异常 + ML概率 + 成本风险矩阵 + 趋势衰减预测
- SHAP 可解释性：每个告警可追溯到具体参数和工业根因（RiskDecomposer → LocalExplainer）
- 四级降级保障：FULL → STAT_ONLY → RULE_ONLY → EMERGENCY，任何条件下不中断输出

**3. 图片占位设计：**
- [中央大幅]：系统架构流程图（从左到右：数据采集→统计基线层→多信号融合层→SHAP解释层→决策输出层，每条线标注对应的技术名词）
- [底部左/中/右]：三个小卡片，分别标注"Z-Score" "T²" "SHAP"，配微型图标

**4. 版式建议：**
中央大图 + 底部三卡片。整体横向流线型布局。架构图用节点+箭头连接，蓝色主色+琥珀色关键节点高亮。本页文字极少，以架构图为核心。

**5. 演讲稿（约 135 字）：**
> 发现瓶颈之后，我们怎么解决？整套架构有四个层次。第一层，逐设备统计基线——每台机床积累3个以上样本后建立自己的"正常画像"，用Z-Score找偏离，用Hotelling T²同时监控四个参数的协变关系。第二层，多信号融合决策——统计异常、机器学习概率、成本风险矩阵、设备趋势衰减，四条独立信号加权融合成最终得分。第三层，SHAP可解释性——每一个告警都必须回答"为什么"：是哪个参数在异常、可能对应什么工业根因。第四层，四级降级——从完整模式到纯规则应急，在任何极端条件下都能产出可执行的维护方案。这不是一个预测模型，这是一个会解释、会降级的工业决策引擎。

**6. 关键数字/关键词：**
per-device baseline · T² multivariate · 4-signal fusion · SHAP · 4-level degradation
